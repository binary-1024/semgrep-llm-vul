# Codex + GPT-5.5 项目开发管理规范,核心不是“让 AI 多写代码”，而是用 **harness 思维**把开发过程变成可约束、可验证、可回滚、可持续演进的系统。

---

# Codex + GPT-5.5 项目开发管理规范

## 1. 总纲

本规范适用于使用 Codex + GPT-5.5 参与软件项目开发、维护、重构、测试、文档和交付的团队。

Codex + GPT-5.5 的最佳使用方式不是把模型当作一次性代码生成器，而是把它放入一个受控的工程开发 harness 中：

> 人类负责目标、边界、判断和验收；  
> Codex 负责探索、实现、验证和迭代；  
> Harness 负责约束、反馈、复现和质量门禁。

本规范的目标是建立一套适合 AI 协作开发的项目方法论，使开发过程具备：

- 明确的任务边界
- 可观察的执行过程
- 可验证的质量结果
- 可回滚的代码变更
- 可沉淀的项目知识
- 可复用的开发流程

---

## 2. 核心思想：Harness-First Development

### 2.1 什么是 Harness 思维

Harness 原意是“挽具、约束装置、测试架”。在 Codex + GPT-5.5 项目开发中，harness 指一整套把模型能力纳入工程系统的机制，包括：

- 任务上下文
- 项目结构
- 代码规范
- 测试体系
- 本地运行脚本
- CI 检查
- Review 标准
- 数据样例
- Mock 环境
- 验收用例
- 回滚策略
- 变更记录

Harness 思维的关键是：

> 不依赖模型“每次都聪明”，而是设计一个环境，让模型即使面对复杂项目，也能持续地产出可检查、可纠偏、可合并的工程结果。

### 2.2 Harness 的四个层级

#### 1. Context Harness：上下文约束

让 Codex 知道：

- 项目是什么
- 当前任务是什么
- 不能改什么
- 应遵循哪些架构边界
- 哪些文件是关键入口
- 当前变更的验收标准是什么

#### 2. Execution Harness：执行约束

让 Codex 的动作可控：

- 使用固定命令启动项目
- 使用固定命令运行测试
- 使用固定命令格式化代码
- 避免破坏用户已有修改
- 只修改任务相关文件
- 避免无关重构

#### 3. Verification Harness：验证约束

让结果可判断：

- 单元测试
- 集成测试
- 类型检查
- Lint
- 构建检查
- UI 截图检查
- 回归用例
- 性能基准
- 安全扫描

#### 4. Feedback Harness：反馈约束

让系统能持续变好：

- 失败日志归档
- 缺陷复盘
- Prompt 模板优化
- 开发规范更新
- 测试用例补充
- 架构说明沉淀

## 3. Codex + GPT-5.5 的角色定位

### 3.1 Codex 适合承担的工作

Codex + GPT-5.5 适合承担：

- 代码阅读和架构理解
- Bug 定位和修复
- 功能实现
- 测试补充
- 重构规划与执行
- 文档生成
- 迁移脚本编写
- UI 原型实现
- API 集成
- Review 辅助
- 复杂任务拆解

### 3.2 Codex 不应独立决定的事项

以下事项必须由人类确认：

- 产品目标变更
- 业务规则解释
- 安全策略调整
- 数据删除或迁移
- 付费服务调用
- 架构方向性重写
- 大规模依赖替换
- 用户隐私相关处理
- 法律、合规、财务相关判断

### 3.3 人机协作原则

Codex 是执行型工程伙伴，不是项目所有者。

人类需要提供：

- 目标
- 优先级
- 业务语义
- 风险边界
- 最终验收

Codex 需要提供：

- 实现方案
- 代码变更
- 验证结果
- 风险说明
- 后续建议

## 4. 项目结构规范

推荐项目包含以下 AI 友好文件：

```text
.
├── README.md
├── AGENTS.md
├── docs/
│   ├── architecture.md
│   ├── development.md
│   ├── testing.md
│   └── decisions/
├── scripts/
│   ├── dev
│   ├── test
│   ├── lint
│   ├── build
│   └── check
├── tests/
├── examples/
├── fixtures/
└── .github/
    └── workflows/
```

### 4.1 `AGENTS.md`

`AGENTS.md` 是 Codex 的项目级操作手册，应包含：

```md
# Agent Instructions

## Project Overview

说明项目目标、技术栈、核心模块。

## Common Commands

- Install: `...`
- Dev: `...`
- Test: `...`
- Lint: `...`
- Build: `...`

## Architecture Boundaries

说明哪些模块负责什么，不允许跨层调用什么。

## Code Style

说明命名、错误处理、日志、测试风格。

## Safety Rules

说明禁止操作，例如不得删除生产数据、不得改动生成文件等。

## Verification

说明每类任务完成前必须运行哪些检查。
```

### 4.2 `scripts/check`

推荐提供一个统一检查入口：

```bash
./scripts/check
```

它应尽可能包含：

- 依赖检查
- 格式检查
- Lint
- 类型检查
- 单元测试
- 构建检查

Codex 最适合在这种统一 harness 下工作。不要让模型猜“这个项目该怎么测”。

## 5. 任务管理规范

### 5.1 任务必须具备清晰边界

每个任务应说明：

- 背景
- 目标
- 非目标
- 涉及模块
- 验收标准
- 风险点
- 是否允许重构
- 是否需要更新测试和文档

推荐格式：

```md
## Task

实现订单导出功能。

## Background

运营需要按时间范围导出订单 CSV。

## Scope

- 新增后端导出接口
- 新增前端导出按钮
- 增加权限校验
- 增加测试

## Out of Scope

- 不修改订单查询逻辑
- 不引入新的报表系统
- 不支持 Excel 格式

## Acceptance Criteria

- 管理员可以导出 CSV
- 普通用户无法访问
- 时间范围参数校验正确
- 测试通过
```

### 5.2 任务粒度

适合 Codex 的任务粒度：

- 1 个明确功能
- 1 个明确 bug
- 1 个明确重构边界
- 1 组相关测试
- 1 篇明确文档

不推荐一次性任务：

- “重写整个系统”
- “优化所有代码”
- “把项目做完整”
- “全面提升质量”
- “随便看看哪里有问题”

复杂任务必须拆成：

1. 探索任务
2. 设计任务
3. 实现任务
4. 验证任务
5. 收尾任务

## 6. Codex 工作流规范

### 6.1 标准工作流

每个开发任务默认遵循：

```text
理解目标
  ↓
阅读相关代码
  ↓
确认实现边界
  ↓
制定简短计划
  ↓
实施代码变更
  ↓
运行验证命令
  ↓
修复失败
  ↓
总结变更和风险
```

### 6.2 不允许的工作方式

Codex 不应：

- 未读代码直接重写
- 为了完成任务大规模重构
- 删除用户已有变更
- 跳过测试直接声称完成
- 引入不必要依赖
- 用临时 mock 掩盖真实问题
- 把业务规则写死在错误层级
- 修改与任务无关的格式或文件

### 6.3 推荐交互方式

推荐：

```text
请阅读当前认证模块，实现 GitHub OAuth 登录。
要求：
- 不改变现有邮箱登录逻辑
- 使用现有 User 模型
- 补充测试
- 完成后运行测试并说明结果
```

不推荐：

```text
帮我加个登录。
```

## 7. 分支与提交规范

### 7.1 分支命名

推荐：

```text
codex/feature-order-export
codex/fix-login-timeout
codex/refactor-payment-service
codex/test-user-permissions
```

### 7.2 提交粒度

每次提交应只表达一个明确意图。

推荐提交信息：

```text
feat: add order CSV export

fix: handle expired login sessions

test: cover payment retry behavior

docs: document deployment process
```

### 7.3 AI 变更必须可审查

AI 生成代码必须满足：

- diff 可读
- 变更范围合理
- 有测试或明确说明无法测试
- 没有无关格式化
- 没有混入临时代码
- 没有隐藏的调试输出

---

## 8. 验证体系规范

### 8.1 每类任务的最低验证要求

#### 后端功能

必须至少运行：

```bash
./scripts/test
./scripts/lint
```

如有类型系统：

```bash
./scripts/typecheck
```

#### 前端功能

必须至少运行：

```bash
./scripts/test
./scripts/lint
./scripts/build
```

涉及 UI 时，应补充：

- 浏览器手动检查
- 截图检查
- 响应式检查
- 空状态、加载态、错误态检查

#### 重构任务

必须运行：

```bash
./scripts/check
```

并说明：

- 外部行为是否改变
- 哪些测试覆盖了行为不变
- 是否存在无法验证的风险

#### 文档任务

至少检查：

- 文档链接有效
- 命令与当前项目一致
- 示例代码可运行或明确标注为伪代码

### 8.2 测试优先级

测试优先覆盖：

1. 业务关键路径
2. 权限和安全边界
3. 错误处理
4. 数据转换
5. 回归 bug
6. 用户可见行为

不要为了覆盖率写无意义测试。

## 9. Review 规范

### 9.1 Code Review 重点

Review AI 代码时，重点看：

- 是否真的满足需求
- 是否改动过大
- 是否破坏架构边界
- 是否遗漏错误处理
- 是否存在安全风险
- 是否引入隐式行为变化
- 是否测试了关键路径
- 是否有不可维护的“聪明代码”

### 9.2 AI 代码常见风险

需要特别警惕：

- 编造不存在的 API
- 误解已有抽象
- 忽略并发问题
- 忽略权限检查
- 忽略事务一致性
- 错误处理过于宽泛
- 用字符串拼接处理结构化数据
- 测试只覆盖 happy path
- UI 看起来完成但状态不完整
- 文档说法和代码不一致

### 9.3 Review 输出格式

推荐：

```md
## Findings

- [P1] 权限校验缺失
- [P2] 测试没有覆盖失败路径

## Questions

- 导出接口是否需要限流？

## Summary

本次变更新增了订单导出能力，但合并前需要补充权限测试。
```

## 10. 适配 GPT-5.5 的使用策略

### 10.1 给模型更多真实上下文，而不是更长指令

GPT-5.5 擅长综合复杂上下文，但前提是上下文真实、准确、可验证。

优先提供：

- 代码入口
- 现有测试
- 错误日志
- 设计文档
- 失败复现步骤
- 目标行为

少用空泛要求：

- “写得优雅”
- “性能好一点”
- “像大厂代码”
- “全面优化”

### 10.2 让模型先读系统，再改系统

对于非小任务，要求 Codex 先做：

```text
先阅读相关代码，说明当前实现路径和你准备修改的文件。
```

但不要让它停留在分析。默认应进入实现和验证。

### 10.3 利用 GPT-5.5 的长程推理能力

适合交给 GPT-5.5 的任务：

- 跨模块 bug 定位
- 架构迁移规划
- 测试体系补齐
- 复杂数据流梳理
- 前后端联动功能
- 老项目现代化改造
- 代码库级别文档生成

但必须配合 harness：

- 明确边界
- 分阶段执行
- 每阶段有验证
- 每次 diff 可审查

### 10.4 避免“无限自主”

不要给 Codex 过于开放的授权：

```text
你看着办，把项目优化一下。
```

更好的方式：

```text
请先扫描项目，列出 5 个最高价值的改进点。不要修改代码。等我确认后再做第一项。
```

或者：

```text
请修复登录超时 bug。可以修改 auth 和 session 相关文件，不要改数据库 schema。完成后运行 auth 测试。
```

## 11. 项目知识沉淀

### 11.1 决策记录

重要决策应写入：

```text
docs/decisions/0001-use-postgres-for-events.md
```

格式：

```md
# Decision: Use PostgreSQL for Event Storage

## Context

## Decision

## Consequences

## Alternatives Considered
```

### 11.2 Insight：决策过程沉淀

`docs/decisions/` 适合记录最终决策，但复杂项目还需要记录“决策是怎么形成的”。

推荐新增：

```text
docs/Insight/
├── README.md
└── templates/
    ├── decision-insight.md
    └── architecture-insight.md
```

`docs/Insight/` 用于记录关键决策前的多视角讨论过程：

- 第一性原理拆解
- 业界做法与备选方案
- 交叉质询与收敛
- 最终决策
- 不采用方案
- 验证方式
- 重新评估条件

通用决策至少从三个视角讨论：

- 程序分析专家
- 工程化专家
- 漏洞安全专家

架构决策至少从两个视角讨论：

- 资深 agent 开发架构师
- 软件工程师

必要时架构决策应补充程序分析专家和漏洞安全专家视角。

适合必须先写 Insight 的情况：

- 项目架构调整
- 数据模型、核心抽象或模块边界变化
- 分析流程、agent 流程或证据链格式变化
- 引入新的程序分析工具、LLM provider、数据库、队列、执行环境或外部服务
- PoC/exp 安全边界变化
- 影响后续多个里程碑的工程决策

推荐关系：

```text
docs/Insight = 记录为什么这样想
docs/decisions = 记录最后决定了什么
```

### 11.3 项目记忆与上下文管理

不要只依赖聊天上下文。对话上下文会变长、压缩和丢失细节，长期有效的信息必须沉淀到项目文件或 Git 历史中。

非小型任务开始前，Codex 应采用分层读取，不要默认读取整个 `docs/`。

第一层，每次优先确认：

- `README.md`
- `AGENTS.md`
- 当前 Git 状态

第二层，按任务类型选择性读取：

- `docs/product.md`
- `docs/methodology.md`
- `docs/roadmap.md`
- `docs/architecture.md`
- `docs/development.md`
- `docs/testing.md`
- `docs/git.md`

第三层，只在涉及具体决策、架构或历史背景时读取：

- 与当前任务相关的 `docs/Insight/<topic>.md`
- 与当前任务相关的 `docs/decisions/<id>.md`

第四层，作为历史材料按需读取：

- 旧 Insight
- 已被取代的 ADR
- 失败复盘或长期归档材料

长文档应优先提供摘要、状态和适用范围，方便 Codex 判断是否需要阅读全文。

推荐文件分工：

- `README.md`：项目入口和文档索引
- `AGENTS.md`：Codex 项目级操作手册
- `docs/product.md`：项目目标、使用场景、输入输出和成功标准
- `docs/methodology.md`：漏洞分析方法论、证据链原则、阶段结论语义和工具/LLM 分工
- `docs/roadmap.md`：阶段路线图
- `docs/architecture.md`：架构说明
- `docs/Insight/`：决策过程
- `docs/decisions/`：最终决策
- `docs/git.md`：Git 管理方式
- `docs/testing.md`：测试策略

`README.md` 应保持稳定，负责说明项目是什么、如何开始、常用命令和核心文档入口。

高频变化的计划、阶段状态和下一步清单应放在 `docs/roadmap.md`，不要直接放在 `README.md` 中，避免多个文档同时维护同一份计划而产生漂移。

如果对话中出现长期有效的新规则、目标或约束，应优先更新项目文档，而不是只保留在聊天记录中。

### 11.4 更适合 Codex 协作的输入方式

为了让 Codex 更稳定地推进项目，用户应尽量提供：

- 当前目标
- 背景和业务语义
- 输入样例
- 期望输出
- 非目标
- 风险边界
- 允许修改的范围
- 验收标准
- 真实错误日志或失败复现步骤

推荐请求方式：

```text
请基于 docs/roadmap.md 的下一个里程碑，先阅读相关文档和代码。
如果涉及架构或核心模型变化，先创建 docs/Insight 记录。
然后实现最小可验证版本，补测试，最后运行 ./scripts/check。
```

不推荐：

```text
你看着办，直接把这个项目做好。
```

用户不需要把所有信息一次说完。更好的方式是让 Codex 用 harness 引导：

1. 先定义目标和边界。
2. 再沉淀文档和 Insight。
3. 然后进入实现。
4. 最后验证、提交和总结。

### 11.5 决策分支与实现分支分离

涉及架构、核心模型、agent 流程、证据链格式、安全边界或跨里程碑影响的任务，推荐拆成两类分支：

- `codex/docs-<topic>`：只做决策、Insight、ADR 和相关文档，不实现代码。
- `codex/feature-<topic>`：基于已经合入 `main` 的决策文档实现代码、fixture 和测试。

推荐流程：

```text
main
  ↓
codex/docs-<topic>       # 完成 Insight / ADR
  ↓
merge back to main
  ↓
codex/feature-<topic>    # 按已合入的决策实现代码
```

这样做的目的：

- 让“为什么这样做”和“具体怎么实现”分开 review。
- 避免在还没完成决策时提前写出难以回滚的实现。
- 让后续实现分支总是基于主干中的最新决策文档。
- 方便重做实现而不丢失决策过程。

### 11.6 失败案例沉淀

每次重大问题应沉淀：

```md
## Incident

## Root Cause

## Missed Signal

## Fix

## New Harness Added
```

关键点是最后一项：

> 每次失败都应该转化成新的 harness。

例如：

- 线上权限 bug → 增加权限测试模板
- 构建失败 → 增加 CI check
- 文档错误 → 增加文档命令验证
- UI 断裂 → 增加截图检查

### 11.7 外部工具 Adapter 的 Fixture Harness

处理 Semgrep、CodeQL、SARIF、Joern、GitHub API、LLM 输出等外部格式时，不能只依赖理想化手写 fixture。

外部工具 adapter 的测试样例应按真实性分层：

- `minimal`：最小合法结构，只覆盖核心字段。
- `realistic`：尽量贴近真实工具输出结构，可以是真实输出的脱敏裁剪版。
- `malformed`：刻意缺字段、结构不完整或格式异常，用于验证错误处理。

Insight 中列出的格式风险和失败模式，必须转化为 fixture 或测试断言。

Review adapter 时必须确认：

```text
这个 fixture 是真实输出、真实输出裁剪版，还是合成的近似结构？
```

如果只能使用合成 fixture，应在 fixture README 或测试中说明依据，并尽快补充 realistic fixture。

### 11.8 单元测试与回归测试规范

项目应明确区分单元测试和回归测试。

单元测试用于验证单个函数、类或小模块的确定性行为：

- 数据模型校验
- 输入解析
- 工具输出归一化
- CLI 参数解析
- 错误处理和边界输入

单元测试要求：

- 不访问网络
- 不依赖真实外部仓库
- 快速、离线、可重复
- 使用 `fixtures/` 或内联小样例
- 新功能必须补单元测试

回归测试用于锁定已经发现过的问题、真实格式差异或历史 bug：

- 已修复 bug
- 外部工具输出格式差异
- realistic fixture
- 安全边界案例
- CLI 行为兼容性

回归测试要求：

- bug fix 必须补回归测试
- 测试名或注释应说明回归点
- 回归 fixture 必须脱敏
- 不能只覆盖 happy path
- Insight 中列出的失败模式应转化为测试

推荐命名：

```text
tests/test_<module>.py
test_<行为>_<预期结果>()
test_regression_<问题摘要>()
```

如果暂时无法补自动化回归测试，必须在交付总结中说明原因和人工验证步骤。

## 12. 交付标准

一个 Codex 参与的任务，只有同时满足以下条件，才算完成：

- 需求已实现
- 代码已通过本地验证
- 关键路径有测试
- 无明显无关改动
- 风险已说明
- 文档已按需更新
- 变更可被人类 review
- 失败或未验证项已明确列出

完成说明推荐格式：

```md
已完成订单 CSV 导出功能。

变更：
- 新增导出接口
- 新增前端导出按钮
- 补充管理员权限测试

验证：
- `./scripts/test` 通过
- `./scripts/build` 通过

风险：
- 暂未加入导出任务队列，大数据量下可能需要异步化
```

## 14. 一句话原则

Codex + GPT-5.5 项目开发的最高原则是：

> 不追求让模型一次性写对所有代码，而是构建一个能让模型持续试错、验证、修正、交付的工程 harness。

好的 AI 开发管理，不是“放权给模型”，而是“把模型接入一个成熟的工程反馈系统”。

当 harness 足够清晰，Codex + GPT-5.5 的能力会从“代码生成”升级为“项目推进”。
