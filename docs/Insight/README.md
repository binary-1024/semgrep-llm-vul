# Insight

`docs/Insight/` 用于记录项目关键决策前的多视角讨论过程。

它和 `docs/decisions/` 的关系：

- `docs/Insight/`：记录决策形成过程，包括第一性原理分析、多角色讨论、备选方案、优缺点和最终建议。
- `docs/decisions/`：记录已经确定的最终决策，采用简洁 ADR 风格，便于长期查阅。

简单说：

```text
Insight = 为什么这样想
docs/decisions = 最后决定了什么
```

## 什么时候必须写 Insight

以下情况必须先写 Insight，再进入实现：

- 项目架构调整。
- 数据模型、核心抽象或模块边界变化。
- 分析流程、agent 流程或证据链格式变化。
- 引入新的程序分析工具、LLM provider、数据库、队列、执行环境或外部服务。
- PoC/exp 安全边界变化。
- 影响后续多个里程碑的工程决策。

以下情况可以不写：

- 小范围文档修正。
- 测试补充。
- 不改变架构的 bug 修复。
- 纯格式化或脚本小修。

## 文件命名

```text
docs/Insight/YYYYMMDD-短标题.md
```

示例：

```text
docs/Insight/20260427-language-agnostic-core.md
docs/Insight/20260427-semgrep-taint-normalization.md
```

## 决策讨论要求

每次重要决策至少包含三轮讨论。

### 通用决策角色

用于产品方向、分析策略、数据模型、证据链和安全能力设计：

- 程序分析专家：关注静态分析、污点传播、调用图、语言差异和可验证性。
- 工程化专家：关注可维护性、测试、模块边界、依赖、性能和交付节奏。
- 漏洞安全专家：关注漏洞语义、攻击面、PoC/exp 安全边界、误报漏报和证据质量。

### 架构决策角色

用于程序架构、agent 架构、模块边界和系统设计：

- 资深 agent 开发架构师：关注 agent 拆分、上下文管理、工具调用、失败恢复和可观测性。
- 软件工程师：关注代码结构、接口稳定性、测试策略、构建部署和长期维护。

架构决策可以同时加入通用决策角色，尤其是涉及程序分析或漏洞验证时。

## 三轮讨论格式

### 第 1 轮：第一性原理拆解

目标：回到问题本质，定义不可妥协的约束。

每个角色回答：

- 这个决策试图解决的根本问题是什么？
- 最小必要能力是什么？
- 哪些约束不能被牺牲？
- 如果做错，最大风险是什么？

### 第 2 轮：业界做法与备选方案

目标：列出主流做法、可选路线和取舍。

每个角色回答：

- 业界通常如何处理类似问题？
- 至少有哪些备选方案？
- 每个方案的优点是什么？
- 每个方案的缺点和失败模式是什么？

### 第 3 轮：交叉质询与收敛

目标：让角色互相挑战，形成可执行决策。

每个角色回答：

- 其他角色的方案里最大风险是什么？
- 哪些假设需要验证？
- 当前阶段最小可行决策是什么？
- 应该如何设计验证 harness？

## 最终决策要求

每份 Insight 最后必须包含：

- 最终决策。
- 为什么现在这样做。
- 不采用哪些方案，以及原因。
- 后续验证方式。
- 何时重新评估。
- 是否需要同步生成或更新 `docs/decisions/`。

## 推荐流程

```text
识别关键决策
  ↓
创建 Insight 记录
  ↓
完成三轮多角色讨论
  ↓
形成最终决策
  ↓
必要时同步 docs/decisions ADR
  ↓
再进入实现
```
