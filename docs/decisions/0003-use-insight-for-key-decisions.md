# Decision: 使用 Insight 记录关键决策过程

## 背景

项目会涉及程序分析、漏洞安全、agent 流程、PoC/exp 安全边界和工程架构等复杂决策。仅在 `docs/decisions/` 记录最终结论，不足以保留形成决策时的推理过程、备选方案和风险权衡。

## 决策

新增 `docs/Insight/` 目录，用于记录关键决策前的多视角讨论。

每次关键决策至少包含三轮：

1. 第一性原理拆解。
2. 业界做法与备选方案。
3. 交叉质询与收敛。

通用决策默认使用三个视角：

- 程序分析专家。
- 工程化专家。
- 漏洞安全专家。

架构决策默认使用两个视角：

- 资深 agent 开发架构师。
- 软件工程师。

必要时架构决策可以补充程序分析专家和漏洞安全专家视角。

## 影响

- `docs/Insight/` 记录决策形成过程。
- `docs/decisions/` 记录最终决策摘要。
- 后续架构、核心模型、agent 流程和安全边界调整，需要先完成 Insight，再进入实现。

## 备选方案

- 只用 `docs/decisions/`：不采用，因为 ADR 适合记录结论，不适合承载完整讨论过程。
- 只保留聊天记录：不采用，因为聊天记录难以检索、复盘和长期维护。
- 引入重型 RFC 流程：暂不采用，因为当前项目仍在早期，轻量 Insight 模板更容易持续执行。
