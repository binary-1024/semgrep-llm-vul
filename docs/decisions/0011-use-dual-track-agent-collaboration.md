# Decision: 使用实现 agent 与研究型 agent 的双轨协作模式

## 背景

`semgrep-llm-vul` 同时依赖工程实现与理论判断。随着调用图、reachability、PoC/exp、安全边界和方法论持续推进，项目需要一种稳定的协作方式，把理论研究与代码落地组织成闭环，而不是在每个关键任务里临时决定“要不要开研究 agent”。

## 决策

项目采用固定的双轨协作模式：

- 主实现 agent 负责主线实现、仓库修改、测试、benchmark、验证、提交和最终交付。
- 研究型 agent 负责理论、算法、论文、行业方案比较、失败模式和阶段性建议。
- 主实现 agent 汇总研究输出与本地实现/验证结果，收敛成下一步 Insight、ADR、任务方案或实现边界。

默认启用双轨模式的任务：

- 架构调整或 agent 流程变化。
- 程序分析算法与能力边界选择。
- 方法论、证据链语义、阶段结论语义调整。
- 会影响多个里程碑的关键工程决策。

默认不必启用的任务：

- 小范围文档修正。
- 已有决策下的直接实现。
- 局部 bug fix、测试补充、fixture 更新。
- 不涉及能力边界变化的日常维护。

## 影响

- 项目的关键算法/架构决策将默认获得理论支撑。
- 研究输出必须沉淀为 Insight、ADR、测试、benchmark 或规范文档，而不是停留在聊天里。
- 最终代码与交付责任仍然集中在主实现 agent，避免多 agent 直接竞争仓库状态。
- 双轨模式会成为后续 M2/M3/M4 推进时的默认协作基线。

## 备选方案

- 单 agent 串行研究与实现：不采用，因为理论探索和主线交付容易互相干扰。
- 对所有任务都强制多 agent：不采用，因为流程成本过高。
- 让研究型 agent 直接承担主线实现：不采用，因为最终验证与集成责任需要单一归口。

## 验证方式

后续通过以下方式验证该决策是否有效：

- 关键任务的 Insight/ADR 是否更完整。
- 研究输出能否稳定转化为代码边界、测试、fixture 或 benchmark。
- 主线任务是否减少因理论边界不清导致的返工或支线膨胀。
- `./scripts/check` 和文档校验持续通过。
