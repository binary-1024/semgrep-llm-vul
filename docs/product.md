# 产品目标与需求

本文档记录 `semgrep-llm-vul` 的项目目标、使用场景、输入输出和阶段性成功标准。

## 项目目标

本项目希望结合程序分析技术与大模型 agent，围绕一个已知或疑似漏洞完成从线索到验证的分析链路：

1. 根据漏洞描述、可能包含漏洞修复的 PR 列表，或用户提供的漏洞代码片段，定位漏洞相关的 sink 函数。
2. 基于 sink 函数，在目标项目中寻找潜在污点路径。
3. 从潜在污点路径中确认可触达污点路径。
4. 基于可触达路径生成 PoC。
5. 进一步生成并执行 exp 脚本，验证漏洞判断是否正确。

项目不是单纯生成代码，而是构建一个可约束、可验证、可追溯的漏洞分析 harness。

## 技术方向

项目采用“语言无关核心模型 + Semgrep 适配层”的方式推进：

- 核心数据模型不绑定具体编程语言。
- Semgrep 作为跨语言扫描和 taint-mode 入口之一。
- 不同语言的差异通过规则、fixture、language profile 和后续分析适配层逐步补充。
- 每个 finding、sink、source、taint path 都保留 `language` 信息，但核心流程不依赖单一语言生态。

第一阶段优先建立统一模型和 Semgrep JSON 归一化能力，而不是直接实现某一种语言的完整漏洞分析。

## 使用场景

### 已知 sink 函数

当用户已经提供 sink 函数时，系统直接从该 sink 出发进行后续分析：

- 解析 sink 函数完整签名。
- 结合漏洞代码片段理解危险行为。
- 在目标项目中寻找 source 到 sink 的污点路径。
- 确认可触达路径。
- 生成 PoC 和 exp 验证脚本。

### 未知 sink 函数

当用户没有提供 sink 函数时，系统需要先挖掘 sink：

- 根据漏洞描述理解风险语义。
- 分析修复 PR 或版本 diff。
- 如果没有修复版本，尽可能定位漏洞引入版本和引入前版本。
- 生成候选 sink 函数，并为每个候选提供证据链。
- 进入污点路径分析、PoC 生成和 exp 验证阶段。

## 输入

### 通用输入

- 目标项目 GitHub repo。
- 受影响版本。
- 修复版本。
- 漏洞描述。

### 可选输入

- sink 函数完整签名。
- 漏洞代码片段。
- 可能包含漏洞修复的 PR 列表。
- 漏洞引入版本。
- 漏洞引入前版本。

### 缺失信息处理

如果没有修复版本：

- 优先尝试定位漏洞引入版本。
- 如果能定位引入版本，则同时获取引入前版本作为对照。
- 如果无法定位，需要明确记录缺失信息和分析风险。

如果没有 sink 函数：

- 先进入 sink 挖掘阶段。
- 生成候选 sink 列表，而不是直接跳到污点路径分析。

## 输出

### 已知 sink 场景

- 污点路径列表。
- 已确认可触达污点路径。
- PoC。
- exp 脚本。
- 每个阶段的完整证据链。

### 未知 sink 场景

- 候选 sink 函数列表。
- 最终确认的 sink 函数。
- 污点路径列表。
- 已确认可触达污点路径。
- PoC。
- exp 脚本。
- 每个阶段的完整证据链。

## 证据链要求

每个阶段输出都必须包含证据链。证据链至少包含：

- 证据来源：漏洞描述、PR、commit、diff、代码位置、调用关系、Semgrep 结果、运行日志等。
- 证据内容：支持当前判断的关键片段或摘要。
- 推理过程：为什么该证据支持当前结论。
- 置信度：当前结论的可信程度。
- 未确认项：还缺什么信息，或者哪些判断仍存在不确定性。
- 可复现方式：如何重新得到该证据，例如命令、文件路径、commit hash、测试步骤。

证据链中的外部数据和源码片段可能包含敏感信息，默认不上传到外部服务。

## 成功标准

项目按四个里程碑推进：

1. sink 函数生成。
2. 污点路径生成与可触达污点路径确认。
3. PoC 生成。
4. exp 验证与生成。

每个里程碑都必须具备：

- 明确输入。
- 明确输出。
- 可复现证据链。
- 自动化或半自动化验证方式。
- 失败时的错误报告和下一步建议。

## 非目标

第一阶段不追求：

- 完整支持所有语言和框架。
- 自动验证所有真实世界漏洞。
- 绕过目标系统防护。
- 生成破坏性利用代码。
- 在没有证据链的情况下给出确定性漏洞结论。

安全边界和合法使用规则需要在进入 PoC/exp 阶段前进一步细化。
